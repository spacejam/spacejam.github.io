property testing

problem: our code reflects our biases, our tests are often biased similarly

solution: don't write tests. write expectations.

have the machine generate random test cases

simplify failing inputs so we can understand

deterministically replay the input as we fix it

But how do we apply random testing to more complex systems?

model-based testing

  1. write a simplified model of a system
  2. apply random sequences of operations on both the implementation and the model
  3. if the model and implementation behave differently, it means we've found a problem
  4. the library will rerun the test, dropping out operations until it finds a minimal failing sequence (the machine just wrote a test for you!!!)

network simulation

  • Jepsen has found bugs in many of the most popular distributed systems
  • but it requires learning clojure and it can take a long time to run
  • if we design our distributed systems to be testable, we can run thousands of partition tests per second
  • by making these tests cheap to run, we can catch race conditions in a second or two that would have been expensive to discover otherwise

lineage-driven fault injection

  • something as complex as a cluster can be challenging to pick the "right" faults for
  • LDFI observes what goes right without faults, then induces targeted faults starting from the last successful operation
  • "Fault-tolerance is redundancy in space and time" - Peter Alvaro
  • when applied to network simulators, this roots out bugs in distributed algorithms extremely effectively

imposing determinism

  • clocks can be controlled externally
  • random number generators can be seeded externally
  • threads can be scheduled using linux realtime priorities
  • files can be wrapped with a mutation log that keeps track of syncs and can be "crashed"
  • check out this crate: deterministic

misc tips

  • use assert! and debug_assert! on non-trivial things! this makes our "fuzzers" extremely effective
  • try not to use unwrap() everywhere, at least use expect() to speed up debugging
  • when propagating errors, include context that helps you get back to the root
